Data Types

üéÆ Video Game Character Database

Imagine designing a database for a fantasy RPG game with millions of players:

CREATE TABLE GameCharacters ( -- Player ID (supports 9 quintillion players!) PlayerID BIGINT PRIMARY KEY, -- Character name (exactly 20 chars, padded with spaces) CharacterCode CHAR(20), -- Character name (variable length, efficient storage) CharacterName NVARCHAR(100), -- Experience points (regular games: INT, but MMOs need BIGINT!) ExperiencePoints BIGINT, -- Gold coins with decimal precision (18 digits, 2 decimal places) GoldCoins DECIMAL(18,2), -- Character creation timestamp CreatedAt DATETIME2, -- Daily login date only LastLoginDate DATE, -- Session time only SessionDuration TIME );
DECIMAL(18,2) for money prevents floating-point errors that could cause players to lose virtual currency! Always use DECIMAL for financial data.

Constraints

üè• Hospital Emergency System

Critical system where data integrity can save lives:

CREATE TABLE Patients ( PatientID INT PRIMARY KEY IDENTITY(1,1), -- Social Security must be unique SSN VARCHAR(11) UNIQUE NOT NULL, -- Emergency contact defaults to hospital EmergencyContact VARCHAR(15) DEFAULT '911-HOSPITAL', -- Age must be realistic Age INT CHECK (Age BETWEEN 0 AND 150), -- Priority level validation Priority VARCHAR(10) CHECK (Priority IN ('LOW', 'MEDIUM', 'HIGH', 'CRITICAL')) ); CREATE TABLE MedicalRecords ( RecordID INT PRIMARY KEY IDENTITY(1,1), PatientID INT FOREIGN KEY REFERENCES Patients(PatientID), Diagnosis NVARCHAR(500) );
CHECK constraints prevent impossible data (like 200-year-old patients) and FOREIGN KEYs ensure no orphaned medical records exist!

Joins

üé¨ Movie Streaming Platform

Netflix-style platform where understanding joins determines what users see:

INNER
Only watched movies
LEFT
All movies + views
RIGHT
All users + movies
FULL
Everything!
-- INNER JOIN: Only movies that have been watched SELECT m.MovieTitle, u.UserName, w.WatchedMinutes FROM Movies m INNER JOIN Watched w ON m.MovieID = w.MovieID INNER JOIN Users u ON w.UserID = u.UserID; -- LEFT JOIN: All movies, show watch data if exists (find unwatched gems!) SELECT m.MovieTitle, u.UserName, ISNULL(w.WatchedMinutes, 0) AS Minutes FROM Movies m LEFT JOIN Watched w ON m.MovieID = w.MovieID LEFT JOIN Users u ON w.UserID = u.UserID; -- SELF JOIN: Find users who watched the same movie SELECT DISTINCT u1.UserName AS User1, u2.UserName AS User2, m.MovieTitle FROM Watched w1 INNER JOIN Watched w2 ON w1.MovieID = w2.MovieID AND w1.UserID < w2.UserID INNER JOIN Users u1 ON w1.UserID = u1.UserID INNER JOIN Users u2 ON w2.UserID = u2.UserID INNER JOIN Movies m ON w1.MovieID = m.MovieID;
INNER = intersection, LEFT = all from left + matching right, SELF JOIN with w1.UserID < w2.UserID prevents duplicate pairs!

WHERE vs HAVING

üçï Pizza Delivery Analytics

WHERE filters individual orders, HAVING filters grouped results:

-- WHERE: Filter individual pizza orders BEFORE grouping SELECT CustomerID, COUNT(*) as TotalOrders, AVG(OrderValue) as AvgValue FROM PizzaOrders WHERE OrderDate >= '2024-01-01' -- Filter individual rows first AND PizzaSize IN ('Large', 'Extra Large') -- Only big pizzas GROUP BY CustomerID HAVING COUNT(*) >= 5 -- Then filter groups (loyal customers) AND AVG(OrderValue) > 25.00; -- High-value customers only -- Think of it as: -- 1. WHERE = "Show me only 2024 large/XL pizza orders" -- 2. GROUP BY = "Group these filtered orders by customer" -- 3. HAVING = "Of these groups, show only customers with 5+ orders averaging $25+"
WHERE happens BEFORE grouping (filters rows), HAVING happens AFTER grouping (filters groups). You can't use WHERE with aggregate functions!

GROUP BY & Aggregates

üéµ Spotify-Style Music Analytics

Analyzing listening patterns across genres and artists:

-- Multiple grouping levels: Genre > Artist > Month SELECT Genre, Artist, YEAR(ListenDate) as Year, MONTH(ListenDate) as Month, COUNT(*) as TotalPlays, -- How many times played COUNT(DISTINCT UserID) as UniqueListeners, -- How many different users SUM(Duration) as TotalMinutes, -- Total listening time AVG(Duration) as AvgSongLength, -- Average song length MIN(Duration) as ShortestSong, -- Shortest track MAX(Duration) as LongestSong -- Epic ballad length FROM MusicStreams WHERE ListenDate >= '2024-01-01' GROUP BY Genre, Artist, YEAR(ListenDate), MONTH(ListenDate) ORDER BY Genre, TotalPlays DESC;
Every column in SELECT must be either in GROUP BY or inside an aggregate function. Think "for each combination of Genre/Artist/Month, calculate these statistics"

CASE Expressions

üèÜ Olympic Games Scoring System

Dynamic categorization and conditional logic for athlete performance:

-- Complex CASE for athlete categorization and medal assignment SELECT AthleteName, Country, Score, -- Medal assignment based on ranking CASE WHEN RANK() OVER (ORDER BY Score DESC) = 1 THEN 'GOLD ü•á' WHEN RANK() OVER (ORDER BY Score DESC) = 2 THEN 'SILVER ü•à' WHEN RANK() OVER (ORDER BY Score DESC) = 3 THEN 'BRONZE ü•â' ELSE 'Participant' END as Medal, -- Performance category CASE WHEN Score >= 9.5 THEN 'World Record Territory' WHEN Score >= 9.0 THEN 'Olympic Record Potential' WHEN Score >= 8.0 THEN 'Strong Performance' WHEN Score >= 7.0 THEN 'Good Effort' ELSE 'Needs Improvement' END as PerformanceLevel, -- Prize money calculation CASE Country WHEN 'USA' THEN Score * 10000 WHEN 'China' THEN Score * 8000 WHEN 'Russia' THEN Score * 7000 ELSE Score * 5000 END as PrizeMoney FROM OlympicResults WHERE Event = 'Gymnastics All-Around' ORDER BY Score DESC;
CASE is like a multi-way IF statement. Use it for categorization, conditional calculations, and transforming data on the fly. Great for creating derived columns!

ORDER BY

üìö Library Book Recommendation Engine

Complex sorting for book recommendations with multiple criteria:

-- Multi-level sorting: Genre preference, then rating, then popularity, then recency SELECT BookTitle, Author, Genre, Rating, ReviewCount, PublicationYear, CASE WHEN Genre IN ('Sci-Fi', 'Fantasy') THEN 1 -- User loves these WHEN Genre IN ('Mystery', 'Thriller') THEN 2 -- Secondary interest ELSE 3 -- Other genres END as GenrePreference FROM Books WHERE Rating >= 4.0 AND ReviewCount >= 100 ORDER BY GenrePreference ASC, -- Preferred genres first Rating DESC, -- Highest rated within each genre preference ReviewCount DESC, -- Most reviewed within same rating PublicationYear DESC, -- Newest books if still tied BookTitle ASC -- Alphabetical as final tiebreaker OFFSET 0 ROWS FETCH NEXT 20 ROWS ONLY; -- Top 20 recommendations
ORDER BY processes in sequence - first by GenrePreference, then Rating within each preference group, etc. Mix ASC/DESC for sophisticated sorting logic!
